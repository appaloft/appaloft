# Dependency Resource Test Matrix

## Scope

This matrix covers the Phase 7 dependency resource lifecycle baseline:

- `dependency-resources.provision`
- `dependency-resources.import`
- `dependency-resources.list`
- `dependency-resources.show`
- `dependency-resources.rename`
- `dependency-resources.delete`
- `resources.bind-dependency`
- `resources.unbind-dependency`
- `resources.rotate-dependency-binding-secret`
- `resources.list-dependency-bindings`
- `resources.show-dependency-binding`
- deployment snapshot safe binding reference capture for active Postgres and Redis Resource bindings
- Redis dependency resource lifecycle
- provider-native Postgres realization scenarios
- provider-native Redis realization scenarios
- dependency resource backup/restore scenarios
- dependency resource scheduled backup policy scenarios

It does not cover provider-native credential rotation, runtime cleanup, backup prune/delete,
backup export, or cross-resource restore.

## Required Non-Coverage Assertions

Tests must assert dependency resource commands do not:

- rotate provider-native database credentials;
- run scheduled backup policy, backup prune/delete, export/download, or cross-resource restore;
- mutate historical deployment snapshots;
- restart, stop, prune, or clean runtime state;
- expose raw connection strings, passwords, tokens, auth headers, cookies, SSH credentials,
  provider tokens, private keys, or sensitive query parameters.

## Current Implementation Notes And Governed Follow-Ups

This baseline implements Postgres dependency resource control-plane records, Resource binding
metadata, safe read models, active-binding delete blockers, and provider-neutral safe deployment
snapshot references for active Postgres and imported Redis bindings. Binding secret rotation is
implemented for binding-scoped safe reference/version metadata only. Redis dependency resource
lifecycle is implemented as provider-neutral safe metadata with imported Redis binding and safe
snapshot reference capture. Neutral vocabulary coverage for `mysql`, `clickhouse`,
`object-storage`, and `opensearch` is implemented for safe read/planning contracts and the Web
selector only; provider-native mutation operations and adapters for those kinds remain governed
follow-up slices. Provider-native Postgres realization is implemented through an injected
provider capability with safe realization state, shell-local realization/delete artifact
materialization, and managed delete cleanup.
Backup/restore is implemented with an injected provider capability for Appaloft-managed Postgres,
Appaloft-managed Redis, and imported external Postgres/Redis metadata, safe provider execution
context (`providerResourceHandle`, masked endpoint, and safe `secretRef`), shell-local native
Postgres command execution for imported Postgres resources with resolvable Appaloft-owned
connection refs, shell-local native Redis logical command execution for imported Redis resources
with resolvable Appaloft-owned connection refs, safe metadata-only artifact materialization for
other references, restore validation, safe backup read models, restore attempt metadata, lifecycle
events, and delete-safety blockers. Dependency binding runtime
injection and store-backed secret value resolution are implemented for imported Postgres, imported
Redis, managed Postgres Appaloft-owned refs, single-server runtimes, Docker Swarm, unresolved-ref
deployment blocking, and retained rotated binding refs through
[Dependency Runtime Secret Value Resolution](../specs/048-dependency-runtime-secret-value-resolution/spec.md).
Provider-native Redis lifecycle is implemented through an injected provider capability for
application-level provision, realization success/failure, ready binding admission, unsupported
provider rejection, shell-local realization/delete artifact materialization, provider cleanup on
delete, persistence/contract safe realization metadata, and managed Redis runtime materialization
through single-server and Swarm secret delivery paths.
Postgres and Redis closed-loop verification are covered by `DEP-RES-PG-CLOSED-LOOP-001` and
`DEP-RES-REDIS-CLOSED-LOOP-001`. The Resource-detail Web console covers provision, external
Postgres/Redis import through the safe connection boundary, rename/delete, backup
create/list/acknowledged restore, bind, list, unbind, and help affordances through
`DEP-RES-WEB-001` and `DEP-RES-BACKUP-011`; broader dependency administration such as scheduled
backup policy, backup prune/delete, export/download, cross-resource restore, and runtime cleanup are
separate governed Phase 7+ capabilities rather than missing pieces of the current baseline.
