## Expected Application Scope

Do not add a public command or query for the first slice.

Add provider-neutral types and services where the edge adapter can call them without importing a
concrete proxy provider:

- `ResourceAccessFailureDiagnostic` value shape;
- `ResourceAccessFailureClassifier` or equivalent service;
- safe renderer input/output types for HTML and problem responses;
- optional short-retention read model port if request diagnostics need owner lookup after the
  immediate response.

The classifier must accept a sanitized provider signal, route context, and request metadata. It must
return a stable diagnostic code/category/phase/retriable/ownerHint tuple without parsing localized
or raw provider text in Web/CLI/API code.

If a request can be tied to a known resource, the application read side may expose the latest safe
edge failure through resource health and diagnostic summary composition. That composition remains
read-only.

## Expected Provider And Adapter Scope

Concrete edge proxy provider packages should map their own gateway failure signals into the
provider-neutral failure envelope. For a Traefik-backed provider this may use:

- gateway status selected by error middleware;
- matched router/service metadata when safe;
- upstream connection error classification;
- route id/resource/deployment labels generated by Appaloft;
- request id propagated through response headers and logs.

Provider packages may use concrete proxy terms internally, but the output to application/Web/API
must be provider-neutral.

The first renderer may be a small HTTP adapter endpoint that Traefik or another edge provider uses
as an error service for gateway-generated 404, 502, 503, and 504 responses.

## Expected Transport Scope

Browser requests:

- return HTML only when the original request accepts HTML;
- include request id, timestamp, code, status, and generic three-hop status;
- reveal owner-only details only after authentication through the console.

API/non-HTML requests:

- return structured problem output;
- include the same code, phase, request id, and retriable flag;
- do not return provider-native raw details.

The diagnostic renderer must not use the command bus, query bus, runtime mutation APIs, or provider
configuration apply paths.

## Expected Web And Desktop Scope

The public page should be product-branded but minimal:

- visitor/browser status;
- Appaloft edge status;
- resource route/upstream status;
- request id and timestamp;
- retry guidance.

Owner surfaces should prefer existing resource detail panels and copyable diagnostics:

- resource health;
- diagnostic summary;
- proxy configuration preview;
- deployment/runtime logs.

User-facing strings in Web/Svelte surfaces must use `packages/i18n` keys. Static HTML produced
outside Web must still avoid hardcoded internal jargon in public text and must preserve the
machine-readable diagnostic code.

## Operation Catalog Scope

No `CORE_OPERATIONS.md` or `operation-catalog.ts` active operation should be added for the edge
diagnostic page itself.

If a later slice introduces a user-callable query such as `resource-access-failures.show`, it must
start with a new Spec Round, update the Business Operation Map, add query/error/testing specs, and
only then update `CORE_OPERATIONS.md` and `operation-catalog.ts`.

## Minimal Deliverable

The minimal Code Round deliverable is:

1. provider-neutral diagnostic type and classifier;
2. concrete edge provider mapping for gateway-generated route/upstream failures;
3. diagnostic renderer for HTML and problem responses;
4. request id propagation into response headers/logs;
5. no replacement of user application responses by default;
6. redaction tests for raw provider/upstream details;
7. resource health and diagnostic summary can include the latest edge failure when safe source data
   exists, or list that as a documented migration gap.

## Required Tests

Required coverage follows [Resource Access Failure Diagnostics Test Matrix](../testing/resource-access-failure-diagnostics-test-matrix.md):

- provider classification for route not found, proxy unavailable, route unavailable, upstream
  unavailable, connect failure, timeout, reset, TLS/protocol failure, and unknown signal;
- HTML versus structured problem rendering;
- user application responses are not replaced by default;
- redaction of raw proxy logs, headers, cookies, private paths, internal addresses, and stack
  traces;
- resource health/diagnostic summary relationship for known resource failures;
- no write side effects or command dispatch.

## Migration Seams And Legacy Edges

Existing resource health and diagnostic summary implementations do not yet read edge request
failure envelopes.

Existing proxy configuration previews can show provider-rendered diagnostics, but they do not
represent per-request gateway failures.

If the first concrete provider uses Traefik error middleware, the middleware/service integration is
an adapter detail. Core/application specs and error codes must remain provider-neutral.

## Current Implementation Notes And Migration Gaps

The first Code Round slice is implemented:

- `packages/application/src/resource-access-failure-diagnostics.ts` defines provider-neutral
  diagnostic types, code/signal parsers, HTTP status fallback mapping, and classification;
- `packages/adapters/http-elysia/src/resource-access-failure-diagnostics.ts` renders HTML or
  problem JSON from safe diagnostic inputs;
- `packages/adapters/http-elysia/src/index.ts` exposes
  `/.appaloft/resource-access-failure`;
- `packages/providers/edge-proxy-traefik/src/index.ts` exposes
  `renderTraefikResourceAccessFailureMiddleware` for provider-rendered error middleware labels;
- Traefik route realization accepts an optional resource access failure renderer target, attaches
  the generated error middleware to served routers, defines the renderer service URL with
  `loadbalancer.server.url`, and leaves redirect-only routers on their redirect middleware;
- while the Appaloft backend service is running, shell composition can expose a provider-neutral
  renderer target to local and SSH runtime backends; wildcard-bound services use
  `http://host.docker.internal:<port>` and the Traefik bootstrap plan adds the matching Docker host
  gateway entry;
- `APPALOFT_RESOURCE_ACCESS_FAILURE_RENDERER_URL` is retained as an optional topology override when
  the proxy manager knows a different reachable backend URL.

Still deferred:

- a companion/static renderer path for one-shot CLI remote SSH execution where no Appaloft backend
  service remains running;
- real Traefik end-to-end probing of the error middleware path;
- request id propagation from the external edge request into the renderer when the provider can
  supply it;
- route/resource/deployment context lookup for owner-facing details;
- short-retention persistence or trace/log lookup for edge failure envelopes;
- `resources.health` and `resources.diagnostic-summary` source composition from latest edge
  failures.

## Open Questions

- Should short-retention edge failure envelopes be persisted in PGlite/PostgreSQL for owner lookup,
  or should logs/traces keyed by request id be the first retrieval path?
- Should public diagnostic pages be customizable per resource or installation, and if yes which
  fields are allowed without creating a new resource access profile command?
